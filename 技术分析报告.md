# GarbageBPFilter v3.0 — 深度技术分析报告

## 一、项目定位

**垃圾 BP 过滤器**是一款面向早期投资机构的 AI 尽职调查工具。核心能力是对初创企业商业计划书（BP）进行自动化事实核查与五维量化评分，识别数据注水与逻辑矛盾，辅助投资人快速筛选项目。

---

| 层级 | 技术选型 |

| 前端 | React 18 + Zustand + Tailwind CSS 3 + Recharts + ECharts + Framer Motion |
| 后端 | Express.js 4.21 + better-sqlite3 (WAL mode) |

## 六、核心业务逻辑

### 6.1 分析流水线（pipelineService.js）

**Step 1 — Agent A 数据提取：**
- 输入：BP原文（PDF解析后的文本）
- LLM 使用系统提示词（prompts.js），从 BP 中提取 11 个标准化指标
- 输出：结构化 JSON（包含声明列表 + 待搜索查询词）

**Step 2 — 并行证据采集：**
- Serper.dev 最多 12 并发搜索，验证 BP 声明
- AkShare 查询行业 PE 基准数据
- 估值背景搜索

**Step 3 — Agent B 校验评分：**
- BP 声明 vs 搜索证据交叉验证（分批处理，每批 6 条声明）
- 结构化评分（团队、产品、商业验证等维度）
- 深度研究报告生成
- 最终通过 scoring.js 计算五维量化分数

### 6.2 五维评分系统（scoring.js）

| 维度 | 权重 | 核心逻辑 |
|------|------|----------|
| S1 时间与天花板 | 20% | `log₁₀(TAM)` 防通胀 + CAGR 线性 |
| S2 产品与护城河 | 20% | TRL(1-9) + 竞争壁垒(1-10) |
| S3 资本效率 | 20% | 行业资本得分 + 规模效应 |
| S4 团队基因 | 20% | 经验年限(对数曲线) + 领域匹配 + 完整度 + 教育 |
| S5 BP诚信度 | 20% | 声明裁决映射(诚实=10, 夸大=3, 证伪=0) |

最终评级：A≥85(直投) / B 75-84(尽调) / C 60-74(观察) / D<60(拒绝)

### 6.3 数据库 Schema 核心表

- **users** — 渐进式注册，含角色、配额、邀请码、头像
- **tasks** — 分析任务全生命周期（UUID主键、状态、进度、结果JSON、文件哈希去重、分享令牌、项目阶段、归档编号、软删除）
- **quotas** — 免费/付费配额分离追踪
- **orders** — 支付订单（幂等 order_no）
- **tokens** — 兑换码系统
- **referrals** — 邀请奖励链

## 七、潜在性能瓶颈

### 7.1 数据库层

| 问题 | 位置 | 影响 | 建议 |
|------|------|------|------|
| **归档编号 LIKE 查询** | taskService.js L24 | `LIKE 'DA-20260405%'` 全表扫描，任务量增长后变慢 | 改用索引数字后缀或 DB sequence |
| **任务缓存无淘汰策略** | taskService.js L244 | 内存 Map 只清理 1 小时前的任务，长时间运行的任务永不释放 | 改用 LRU 缓存或按最后访问时间淘汰 |
| **getStats 全量 JSON 解析** | userController.js L165 | 为每个任务解析 result JSON（可能 30KB+），即使只需统计数 | 在 DB 层聚合或缓存统计结果 |
| **WAL 文件无限增长** | db/index.js | 高并发写入下 WAL checkpoint 策略缺失 | 添加定期 checkpoint PRAGMA |
| **busy_timeout 仅 5s** | db/index.js L27 | 高并发下仍可能触发 "database locked" | 应用层添加重试策略 |

### 7.2 LLM 调用层

| 问题 | 位置 | 影响 | 建议 |
|------|------|------|------|
| **重试无抖动** | llmService.js L69 | 指数退避无随机抖动，并发请求同时重试造成惊群效应 | 添加 jitter: `delay * (0.5 + Math.random())` |
| **批量校验串行化** | pipelineService.js | 声明分批（每批6条）串行处理，100条声明需17次LLM调用 | 可并行处理多个批次 |
| **Promise.all 不 fail-fast** | pipelineService.js L197 | 结构化评分+深度研究并行，一个失败另一个仍在浪费资源 | 改用 Promise.allSettled 或 AbortController |
| **超时公式过于保守** | llmService.js L23 | maxTokens=12000 时总超时 480s，但实际推理可能更长 | 根据历史数据动态调整 |

### 7.3 前端层

| 问题 | 位置 | 影响 | 建议 |
|------|------|------|------|
| **历史页无分页** | HistoryPage.jsx | 所有任务一次加载渲染，1000+任务时 DOM 爆炸 | 实现分页或虚拟滚动 |
| **轮询间隔固定 2.5s** | useAnalysisPipeline.js | 分析初期频繁轮询浪费带宽，后期又响应不够快 | 使用自适应退避策略 |
| **声明过滤每次渲染重算** | DetailedReport L147 | 6 次 filter 操作在每次渲染时重复执行 | 使用 useMemo 缓存统计结果 |
| **大结果对象驻留 Zustand** | useAnalysisStore.js | 30KB+ 分析结果始终在内存中 | 按需加载或分片存储 |
| **无请求去重/缓存** | api.js | 相同 API 重复调用不缓存 | 添加请求级缓存 |

## 八、潜在逻辑漏洞

### 8.1 严重（CRITICAL）

1. **文件去重竞态条件** — `analyzeController.js L84-92`
   - 问题：两个相同文件的请求同时到达，两者都通过 `findRunningTask()` 检查（均返回 null），创建重复任务
   - 影响：浪费 LLM 配额，结果可能冲突
   - 修复：使用数据库 UNIQUE 约束 + INSERT OR IGNORE 实现原子去重

2. **配额扣减时序漏洞** — `analyzeController.js + quota.js`
   - 问题：`checkQuota()` 在中间件检查通过后，`deductQuota()` 在任务创建后才执行。两个请求可能同时通过检查但配额只够一次
   - 影响：配额超支
   - 修复：将"检查+扣减"合并为单个原子事务操作

3. **归档编号生成竞态** — `taskService.js L24-26`
   - 问题：两个并发 `createTask` 可能查到相同的最大编号，生成重复的 `DA-YYYYMMDD-XXXX`
   - 影响：归档编号冲突
   - 修复：对 archive_number 添加 UNIQUE 约束，或使用事务内递增

### 8.2 高危（HIGH）

4. **配额退还静默失败** — `analyzeController.js L212-214`
   - 问题：分析失败时 `refundQuota()` 如果本身抛出 DB 错误，异常被吞没，用户配额永久丢失
   - 修复：添加重试机制 + 错误日志告警

5. **月度统计除零错误** — `userController.js L259`
   - 问题：`lastMonth.count === 0` 时，环比增长率计算 `(thisMonth - lastMonth) / lastMonth` 产生 Infinity
   - 修复：添加零值保护，返回 null 或特殊标记

6. **软删除不一致** — `taskService.js`
   - 问题：`getTask()` 不过滤 `deleted_at`，但 `getTasksByUser()` 过滤了。通过 taskId 直接访问可看到已"删除"的任务
   - 影响：数据泄露
   - 修复：所有查询统一添加 `WHERE deleted_at IS NULL`

7. **客户端断开后流水线继续运行** — `analyzeController.js L145-216`
   - 问题：异步分析块不监听 `req.on('close')`，客户端断开后 LLM 调用继续执行
   - 影响：浪费计算资源和 LLM 配额
   - 修复：添加 AbortController，在连接断开时取消 LLM 请求

### 8.3 中危（MEDIUM）

8. **批量校验失败降级不彻底** — `pipelineService.js L169`
   - 问题：批量校验失败后的单条重试 `verifySingleClaim()` 本身没有重试循环，如果也失败则永久标记为"存疑"
   - 影响：分析准确性下降

9. **分享链接时区不一致** — `taskController.js L50`
   - 问题：过期时间使用 `new Date()` 比较，但数据库存储的字符串可能无时区标记（缺少 Z 后缀）
   - 影响：分享链接可能提前或延迟过期

10. **PDF 下载报告 HTML 注入** — `downloadReport.js L30`
    - 问题：分析结果中的 `industry_name` 等字段直接插入 HTML 模板，未转义
    - 影响：如果 LLM 返回包含 `<script>` 的内容，存在 XSS 风险

11. **邮箱注册竞态** — `authController.js L165-171`
    - 问题：检查邮箱是否已注册与实际注册不在同一事务内，两个请求可能同时通过检查
    - 修复：依赖数据库 UNIQUE 约束 + 捕获 IntegrityError

## 九、安全风险

| 风险 | 严重度 | 说明 | 建议 |
|------|--------|------|------|
| **JWT Secret 弱密钥** | 严重 | .env.example 含占位符；dev 模式每次重启随机生成（使旧 token 全部失效） | 生产环境强制 48+ 字符随机密钥 |
| **Token 存储在 localStorage** | 高 | 前端 `bp_token` 存于 localStorage，XSS 即可窃取 | 改用 httpOnly Secure Cookie |
| **无 CSRF 保护** | 高 | fetch 调用未携带 CSRF Token | 添加 X-CSRF-Token 头或 SameSite Cookie |
| **SQLite 无加密** | 中 | 数据文件明文存储在 Docker 卷 | 评估 SQLCipher 加密 |
| **Admin 密码明文配置** | 中 | ADMIN_PASSWORD 在 .env 中明文 | 使用 Secrets Manager |
| **上传文件无病毒扫描** | 中 | PDF/PPTX 直接传给 OCR 引擎 | 添加 ClamAV 扫描 |
| **开放重定向** | 低 | LoginForm 的 redirect 验证不充分 | 白名单校验路由 |

## 十、架构改进建议

1. **引入消息队列**：将分析任务投入 Redis/BullMQ 队列，解耦请求处理与 LLM 调用，支持横向扩展
2. **API 版本化**：当前所有端点 `/api/X`，schema 变更会破坏客户端。建议 `/api/v1/X`
3. **输入校验框架**：使用 Zod/Joi 统一校验请求体，替代分散在 Controller 中的手动检查
4. **缓存层**：高频读取的用户信息、配额状态应添加 Redis 缓存
5. **日志结构化**：当前 logger 缺少请求上下文自动注入，分布式追踪困难
6. **TypeScript 迁移**：前后端均为纯 JS，缺乏类型安全，重构时易引入 bug
7. **测试覆盖**：目前仅有 `jsonParser.test.js` 和 `scoring.test.js`，核心流水线无测试覆盖
8. **SSE 替代轮询**：前端每 2.5s 轮询任务状态，改用 Server-Sent Events 可减少 80%+ 无效请求
