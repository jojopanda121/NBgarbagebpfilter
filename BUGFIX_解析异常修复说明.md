# 解析异常问题修复说明

## 问题描述
在测试过程中，分析 BP 时总是显示"解析异常"，尝试 3-4 次都无法成功生成报告。

## 根本原因分析

经过代码审查，发现以下几个导致解析失败的关键问题：

1. **LLM 输出格式不稳定**
   - MiniMax 模型可能在 JSON 前后添加解释性文字
   - 可能使用 markdown 代码块包裹 JSON
   - 可能包含注释（单行 `//` 或多行 `/* */`）
   - 可能有尾逗号（trailing comma）

2. **JSON 提取逻辑不够健壮**
   - 正则表达式匹配不全面
   - 没有处理嵌套大括号的情况
   - 缺少多种候选方案的尝试

3. **错误处理不完善**
   - 解析失败时缺少详细的调试信息
   - 只重试 1 次，成功率不高
   - 没有对 LLM 输出格式进行更严格的约束

## 修复方案

### 1. 增强 JSON 提取函数 (`extractJson`)

**改进点：**
- ✅ 支持多种 markdown 代码块格式（`json`、无标签、多个反引号）
- ✅ 精确匹配嵌套大括号（使用括号计数算法）
- ✅ 增加多个候选方案（6 种提取策略）
- ✅ 每个候选尝试 4 种解析方法：
  1. 直接解析
  2. sanitize 后解析（移除注释、尾逗号）
  3. 修复常见问题后解析（BOM、零宽字符等）
  4. 组合修复
- ✅ 详细的错误日志，便于调试

**代码位置：** `server/index.js` 第 195-270 行

### 2. 改进 JSON 清理函数 (`sanitizeJsonString`)

**新增功能：**
- ✅ 移除多行注释 `/* ... */`
- ✅ 移除 BOM 和零宽字符
- ✅ 更好的空白字符处理

**代码位置：** `server/index.js` 第 186-193 行

### 3. 新增 JSON 修复函数 (`attemptJsonFix`)

**功能：**
- ✅ 移除 BOM 和零宽字符（`\uFEFF`, `\u200B-\u200D`）
- ✅ 修复尾逗号问题
- ✅ 移除所有类型的注释
- ✅ 规范化空白字符

**代码位置：** `server/index.js` 第 195-210 行

### 4. 强化系统提示词

**Agent A (诉求提取器) 改进：**

- ✅ 禁止使用 markdown 代码块

- ✅ 禁止前后添加解释文字
- ✅ 提供完整的 5 个维度示例

**Agent B (法官裁决者) 改进：**
- ✅ 同样的格式约束
- ✅ 强调字符串转义规则
- ✅ 明确 JSON 结构要求

**代码位置：** `server/index.js` 第 233-313 行

### 5. 增加重试机制

**Claim 提取阶段（第 1 步）：**
- ✅ 首次失败后自动重试
- ✅ 重试时添加更严格的提示词
- ✅ 验证 claims 数组的完整性

**代码位置：** `server/index.js` 第 401-550 行

**新增日志内容：**

- ✅ 失败时的调试提示

## 使用建议

1. **如果仍然遇到解析失败：**
   - 查看服务器日志（终端输出）
   - 检查 `[extractJson]` 开头的日志
   - 查看原始输出内容，判断是否为 API 返回异常

2. **如果问题持续：**
   - 检查网络连接
   - 确认 MINIMAX_API_KEY 是否有效
   - 查看 API 配额是否用尽
   - 尝试更换 PDF 文件测试

3. **优化建议：**
   - 上传文字清晰的 PDF（避免扫描版）
   - BP 内容尽量完整（至少包含市场、团队、产品等信息）
   - 文件大小控制在 10MB 以内

## 技术细节

### JSON 提取策略优先级

1. **代码块提取**（优先级最高）
   - ````json ... ```
   - ``` ... ```
   - 多反引号变体

2. **大括号边界提取**
   - 最外层 `{...}` 
   - 第一个完整对象（括号匹配）

3. **原始文本**（兜底）

### 解析尝试顺序

对每个候选方案：
1. 直接 `JSON.parse()`
2. `sanitizeJsonString()` 后解析
3. `attemptJsonFix()` 后解析
4. 组合修复后解析

### 重试策略

- **Claim 提取：** 2 次尝试（原始 + 严格提示）
- **裁决分析：** 3 次尝试（thinking + 普通 + 简化）
- **总计：** 最多 5 次 LLM 调用（正常情况 2-3 次）
