   - `443` (HTTPS)
   - `22` (SSH)
   - `3001` (应用端口)

## 七-A、代码更新指南（git pull 安全说明）

### 安全更新步骤

```bash
ssh root@你的服务器IP
cd /root/NBgarbagebpfilter

# 第1步：备份数据库（可选但推荐）
cp data/app.db "data/backups/app.db_$(date +%Y%m%d_%H%M%S).backup.db"

# 第2步：拉取最新代码
git pull

# 第3步：重新构建并启动（数据库不受影响）
docker-compose up -d --build
```

### 为什么数据库不会丢？

1. 数据库文件在宿主机 `./data/app.db`，不在 Docker 容器内
2. `git pull` 不会覆盖数据库——`.gitignore` 排除了所有 `.db` 文件
3. `docker-compose up --build` 重建容器时，`./data` 通过 bind mount 挂载，容器销毁重建不影响宿主机文件
4. 新的数据库迁移会自动执行，已执行过的迁移会跳过（通过 `schema_migrations` 表追踪）

### 绝对不要做的事

| 危险操作 | 后果 |
|---------|------|
| `docker-compose down -v` | 删除所有数据卷 |
| `rm -rf data/` | 直接删除数据库 |
| 手动修改 `data/app.db` 的文件权限为不可写 | SQLite 无法写入 |
| 在 `.env` 中修改 `DB_PATH` 指向新路径 | 旧数据库不会被读取 |

3. 使用 `docker-compose --profile production up -d` 启动 Nginx
4. 配置SSL证书（参考"九、下一步"）

### Q4: 数据会丢失吗？
不会！数据库保存在 `/root/NBgarbagebpfilter/data` 目录

## 十二、数据库备份与恢复（非常重要！）

### ⚠️ 数据丢失问题原因

**问题根源**：如果你在更新时执行了 `docker-compose down -v`，数据会被删除！

**解决方案**：
1. 永远不要执行 `docker-compose down -v`（-v 参数会删除所有数据卷！）
2. 更新正确方式：`git pull && docker-compose up -d --build`
3. 定期备份数据库

### 手动备份

```bash
cd /root/NBgarbagebpfilter

# 创建备份目录
mkdir -p data/backups

# 备份数据库
cp data/app.db "data/backups/app.db_$(date +%Y%m%d_%H%M%S).backup.db"

# 备份WAL文件（如果存在）
[ -f data/app.db-wal ] && cp data/app.db-wal "data/backups/"
[ -f data/app.db-shm ] && cp data/app.db-shm "data/backups/"

# 查看备份
ls -la data/backups/
```

### 自动备份脚本

项目已内置备份脚本：

```bash
# 本地备份
./scripts/backup-db.sh

# 安装定时任务（每天凌晨3点自动备份）
./scripts/backup-db.sh --install-cron
```

### 备份到腾讯云COS（可选）

```bash
# 安装coscmd
pip install coscmd

# 配置COS
export COS_BUCKET=your-bucket
export COS_SECRET_ID=your-secret-id
export COS_SECRET_KEY=your-secret-key
export COS_REGION=ap-guangzhou

# 上传备份
./scripts/backup-db.sh --cos
```

### 数据恢复

如果数据丢失，可以从备份恢复：

```bash
# 1. 停止服务
docker-compose down

# 2. 找到最近的备份文件
ls -lt data/backups/

# 3. 恢复数据库
cp data/backups/app.db_xxxxxx.backup.db data/app.db

# 4. 恢复WAL文件（如果备份中包含）
cp data/backups/app.db_xxxxxx.backup.db-wal data/app.db-wal 2>/dev/null || true
cp data/backups/app.db_xxxxxx.backup.db-shm data/app.db-shm 2>/dev/null || true

# 5. 重启服务
docker-compose up -d
```
